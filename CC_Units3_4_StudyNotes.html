UNIT 3

1. Storage Types & Architectures

Three Types of Data Storage

TypeHow it worksExample
File StorageOrganises data in a hierarchy of files and folders. Computer needs the full path to access data. Uses metadata to locate files.NAS, DAS
Block StorageChops data into fixed-size blocks, each with a unique ID. Storage system reassembles blocks on request. No metadata handling.SAN environments
Object StorageData stored as self-contained objects in a flat namespace. Each object has: data + unique ID + custom metadata. Accessed via REST APIs.Amazon S3, Azure Blob
Block storage is expensive and has limited metadata capability. Object storage uses GET/PUT/DELETE REST APIs.

Storage Architectures

  • DAS (Directly Attached Storage): Storage physically connected to one server. Simple but not shareable.
  • NAS (Network Attached Storage): File-level storage accessible over a network. Supports multiple users simultaneously.
  • SAN (Storage Area Network): Dedicated high-speed network connecting servers to block-level storage devices.

Magnetic Disk Latency

Disk Latency = Seek Time + Rotational Latency + Transfer Time
  • Seek time: Head moves to the correct cylinder/track
  • Rotational latency: Wait for the right sector to spin under the head
  • Transfer time: Actual read/write of data

RAID

Redundant Array of Independent Disks — combines multiple physical disks into one logical unit for redundancy + performance. Data distributed across drives in different "RAID levels" (RAID 0, 1, 4…)

2. Cloud Storage & Enablers

What is Cloud Storage?

A data storage model where data is maintained and backed up remotely, accessible over the internet. The provider manages the physical environment; users buy/lease capacity.

8 Key Properties of Cloud Storage

  1. Dramatic TCO reduction – 10× cheaper than block/file storage
  2. Unlimited scalability – built on distributed tech; add/remove nodes seamlessly
  3. Elasticity – virtual pool grows/shrinks as needed
  4. On-demand / pay-as-you-go – pay only for what you store & access
  5. Universal access – no SAN attachment required
  6. Multi-tenancy – shared, centrally managed, cost-efficient
  7. Data durability & availability – commodity hardware + software HA layer
  8. Disaster Recovery

Cloud Storage Enablers

File Systems

Controls how data is stored/retrieved. A file system provides structure (without it, storage is just a blob of data). Block = basic I/O unit.

Network File Systems (NFS): Makes local files available over LAN. Server exports directories via nfsd daemon; clients mount them — transparently accessible.

LVM – Logical Volume Manager

  • Independent layer between file system and physical disks
  • Physical Partitions → Physical Volumes → Volume Group → Logical Volumes → File System
  • Allows dynamic resizing of storage without reformatting

Thin Provisioning (Virtual Provisioning)

Hosts are presented with more capacity than is physically allocated. Physical storage allocated only when actually needed. Allows provisioning decisions without worrying about currently available storage.

Thin provisioning = logical capacity > physically allocated capacity

3. Storage Virtualization

What is Storage Virtualization?

Abstracts physical storage subsystems from applications and presents them as logical entities — hiding complexity of physical devices, network changes, and access types.

Can: aggregate multiple devices into one pool OR split one large device into many smaller logical entities.

Category 1: File-Level Virtualization

Eliminates dependency between file access location and physical storage location. Manages files/directories across multiple servers as a single logical file system.

Distributed File System (DFS)

  • File system distributed across multiple servers
  • Provides location transparency + fault tolerance + replication
  • Two metadata management approaches:
ApproachSystemHow it works
Centralised metadataLustreDedicated MDS (Metadata Server). Good for large files. MDS can become a bottleneck under heavy metadata ops. Clients → MDS for layout → OSTs for actual data.
Distributed metadataGlusterFSNo central metadata server. Metadata spread across all nodes. Uses "translators" (stackable filter programs) for features like AFR (replication), I/O scheduler. Higher complexity.

Category 2: Block-Level Virtualization

Virtualises multiple physical disks → single logical disk. Block addresses may span multiple physical subsystems.

LevelHow
Host-basedUses LVM to create storage pool from multiple disks
Storage-device levelRAID in array controllers creates LUNs across disks. Low latency, host-agnostic.
Network-basedMost scalable. In Fibre Channel SAN. Two types: Switch-based (virtualisation in switch + metadata manager) or Appliance-based (I/O flows through appliance). Appliance can be in-band (all I/O through it) or out-of-band (only metadata through it, data path is direct).

4. Object Storage & Amazon S3

Object Storage Basics

Stores data as flat-namespace objects (not files in folders or blocks). Each object = data + unique ID + metadata. Uses REST HTTP APIs (GET, PUT, DELETE).

Built on scale-out distributed systems using commodity hardware. Key tasks: data placement + durability automation.

Object storage uses REST APIs. Objects have unique IDs, no directory hierarchy (flat namespace).

Amazon S3

  • Buckets: containers for objects. Up to 100 buckets/account, unlimited objects/bucket.
  • Objects: up to 5 TB each. Replicated across multiple geographic locations.
  • Keys: path-like name for an object. URL: http://s3.amazonaws.com/bucketname/key
  • Security: Access control policies (via AWS Console) + Audit logs (stored in separate bucket)
  • Data Protection: Default replication across multiple devices (survives 2 replica failures). Reduced Redundancy Storage (RRS) = 2 replicas, survives 1 failure.
  • Regions: Choose at bucket level for performance/legal reasons
  • Versioning: Stores full history; allows undo deletes
  • Multi-part Uploads: Split large objects, upload parts in parallel; only failed parts need retry

OpenStack Swift

Another object storage system. Key concepts: Account (user in storage system) → Container (namespace to group objects) → Object (actual data). Partitions = storage locations distributed across disks. Ring = maps partition space to physical disk locations (like a hash-based encyclopedia).

5. DynamoDB (NoSQL)

DynamoDB Key Concepts

  • Fully managed NoSQL key-value + document database on AWS
  • Tables, Items, Attributes are core components
  • Supports only item-level consistency (row-level, no cross-item transactions)
  • No joins between tables — application must handle join logic

Primary Key Types

  • Partition key only: Value is hashed → determines physical partition
  • Partition key + Sort key (Composite): Items with same partition key stored together, sorted by sort key

Secondary Indexes

Allow querying by non-primary key attributes. Partitions are automatically replicated across multiple Availability Zones.

DynamoDB uses hash function on partition key to determine storage location. No joins supported.

6. Partitioning & Consistent Hashing

Partitioning

Intentionally splitting large data into smaller pieces (partitions), each on a single node or distributed across nodes — enables parallel queries and better throughput.

  • Skew: uneven distribution across partitions
  • Hot spot: one partition with disproportionately high load

4 Approaches to Partitioning

TypeIdeaPros/Cons
VerticalDifferent columns on different nodesImproves performance for column access patterns
Workload-drivenPartition based on application access patternsImproves transaction scalability
Random assignmentAssign records randomly to nodesEven distribution but requires querying all nodes for reads
Horizontal (key-based)Static: by key range, schema, graph, or hashMost common; various trade-offs below

Partitioning by Key Range

Each partition gets a continuous range of keys. Easy range queries (e.g., all sensor readings in January). Risk: timestamp keys → all writes go to today's partition → hot spot.

Partitioning by Hash of Key

Hash function evenly distributes rows across partitions. Eliminates skew and hot spots. Downside: loses ability to do efficient range queries. Composite keys (e.g., user_id + timestamp) can partially restore range query ability.

Distributed Hashing & the Rehashing Problem

Simple hash(key) mod N assigns keys to N servers. Problem: when a server is added/removed, almost all keys need to be remapped → massive data movement.

Consistent Hashing ⭐

Maps both keys and servers onto a virtual circle (hash ring). Each key is assigned to the nearest server in the clockwise direction.

Key benefit: When a server is added or removed, only the keys closest to it (clockwise) need to be moved. The rest remain unaffected. Scales without affecting the overall system.
Consistent hashing minimises key redistribution when servers join/leave. Used in distributed hash tables.

Secondary Indexes in Partitioned DBs

TypeHow it worksTrade-off
Document-based (local index)Each partition maintains its own secondary index for its documents onlyWrites are fast (local only). Reads require scatter/gather across all partitions → expensive
Term-based (global index)One global index covering all partitions, itself partitioned by termReads are efficient (query one partition). Writes are slow/complex (may update index in multiple partitions; often async)

Rebalancing Partitions

Moving load between nodes when: throughput increases, dataset grows, a machine fails, or a machine rejoins. Requirements: fair load after rebalancing, DB stays available during rebalancing, minimum data movement.

Strategies

  • hash mod N: Simple but moves most keys on any change — too expensive
  • Fixed number of partitions: Create more partitions than nodes; new nodes steal some. Easy to manage but hard to tune partition size for variable datasets.
  • Dynamic partitioning: Partition splits when it exceeds a size threshold; merges when it shrinks below. Number of partitions adapts to data volume. Starts slow (single partition at beginning).
  • Proportional to nodes: Fixed partitions per node. When nodes increase, each partition gets smaller. Uses random splitting.

Request Routing

  • Approach 1: Any node → forwards to correct node if needed (round-robin)
  • Approach 2: Routing tier acts as partition-aware load balancer
  • Approach 3: Client is partition-aware, connects directly
  • Approach 4 (ZooKeeper): External coordination service tracks partition-to-node mapping. Nodes register themselves; routing tier subscribes to changes.
HBase, Kafka, SolrCloud use ZooKeeper for partition tracking. Cassandra/Riak use gossip protocol instead.

7. Replication

Why Replicate?

  • Reduce latency (geographically close replicas)
  • Increase availability (system survives partial failures)
  • Increase read throughput (more nodes can serve reads)

Algorithm 1: Leader-Based (Single-Leader) Replication

One replica = leader (master/primary). Clients write only to the leader. Leader writes locally, then sends replication log to followers. Clients can read from any replica.

Sync vs Async Replication

SynchronousAsynchronous
When does leader respond?After follower confirms receiptImmediately, without waiting for follower
GuaranteeFollower always has latest dataFollower may lag behind
DownsideWrite blocked if follower is slow/downReplication lag → stale reads

Replication Log Implementations

  1. Statement-based: Log every SQL/write statement. Problem: non-deterministic functions (NOW(), RAND()) differ on replicas.
  2. Write-ahead log (WAL) shipping: Append-only byte sequence sent to followers. Tied to storage engine version.
  3. Change Data Capture (logical log): Row-level changes. Engine-agnostic. Easier for external tools to parse.
  4. Trigger-based: Application code fires on data change; external process replicates. Flexible but higher overhead.

Failure Handling

  • Follower failure: Uses local log to determine last processed transaction → requests missing changes from leader (catch-up recovery)
  • Leader failure (Failover): 1) Detect leader failure (timeout) 2) Elect new leader (usually most up-to-date follower) 3) Reconfigure clients to use new leader

Replication Lag

Delay between write on leader and it being reflected on follower. In normal operation: fraction of a second. Under load: can grow significantly → stale reads.

Eventual consistency: If no new writes, followers will eventually converge to leader's state.

Solutions: Always read critical data from leader; monitor lag on followers; client remembers its latest write timestamp; monotonic reads (same user always reads from same replica); consistent prefix reads.

Algorithm 2: Multi-Leader Replication

Multiple nodes accept writes. Each leader replicates to other leaders and their followers. Also called master/master or active/active.

Use cases: Multi-datacenter (one leader per DC), clients with offline operation (device = local leader), collaborative editing.

Main challenge: Write conflicts. Solutions: conflict avoidance (route same record to same leader), last-write-wins (highest ID wins), custom conflict resolution logic.

Topologies: Circular, Star, All-to-all (all-to-all is most fault-tolerant — no single point of failure).

Single-LeaderMulti-Leader
LatencyAll writes go to one DCWrites processed locally per DC
DC outageFailover promotes new leaderOther DCs keep operating independently
Network problemsSensitive (writes are synchronous)Tolerant (async replication)
ConflictNoneMust handle write conflicts

Algorithm 3: Leaderless Replication

All nodes accept writes and reads. No designated leader. Client broadcasts writes to all replicas, considers write successful after k-of-n ACKs (quorum).

Quorum rule: w + r > n ensures reads always see at least one up-to-date write.
Common: n odd (3 or 5), w = r = (n+1)/2

Handling stale replicas

  • Read repair: Client detects stale replica during read, writes newer value back
  • Anti-entropy: Background process compares replicas and copies missing data

Concurrent Write Detection

  • Last write wins (LWW): Highest timestamp/version number survives. Problem: clock sync issues.
  • Version vectors: Per-replica version numbers to detect concurrency and merge values
  • Happens-before: Operation A happens before B if B knows about A (depends on A)
Leaderless replication: better availability (no single bottleneck). Quorum: w + r > n. Used in Dynamo/Cassandra-style systems.

8. Consistency Models & CAP Theorem

Consistency Models

ModelGuarantee
Strict / LinearizabilityRead always returns most recent write. Single-copy illusion. Operations atomic. Also: Strong consistency, Immediate consistency.
Sequential ConsistencyAll processes see operations in the same order, but not necessarily real-time order.
Causal ConsistencyOnly causally related operations must be seen in same order. Unrelated operations can be seen in different orders.
PRAM ConsistencyAll writes by a single process seen by all others in the same order (like a pipeline). Writes by different processes may be seen differently.
Eventual ConsistencyIf no new writes, replicas will eventually converge. Very weak guarantee — no bound on when.

Linearizability

Basic idea: system appears as if there's only ONE copy of data, all operations atomic. Once any client reads new value after a write, all subsequent reads must return that new value. Compare-and-Set (CAS) is a key operation: atomically update if current value matches expected.

  • Single-leader replication with synchronous followers → potentially linearizable
  • Consensus algorithms (ZooKeeper) → linearizable
  • Multi-leader, Leaderless → NOT linearizable

CAP Theorem ⭐

In a distributed system with data replication, you can only strongly guarantee 2 of 3:

  • C – Consistency: All nodes return the same, most recent data at any time
  • A – Availability: Every request gets a response (not an error)
  • P – Partition Tolerance: System keeps running despite network partitions
Since network partitions (P) are unavoidable in real cloud systems, the real trade-off is between C and A:
  • CP systems: Consistent + Partition Tolerant. Sacrifice availability. E.g., PostgreSQL, ZooKeeper
  • AP systems: Available + Partition Tolerant. Sacrifice strict consistency (use eventual consistency). E.g., Cassandra, CouchDB

Criticisms of CAP: Ignores latency; reality is a continuum not binary; "pick two" is only relevant during a partition — when there's no failure, you can have both C and A.

CAP: can't have all 3. Since P is unavoidable → choose CP or AP. Cassandra = AP. PostgreSQL = CP.

9. Distributed Transactions & 2PC

ACID Properties

PropertyMeaning
AtomicAll-or-nothing: either all steps complete or none do. No partial results visible.
ConsistentSystem invariants hold before and after. (e.g., total money in all accounts stays same)
Isolated (Serializable)Concurrent transactions produce same result as if they ran serially
DurableCommitted results are permanent; no failure can revert them
ACID = Atomic, Consistent, Isolated, Durable. All 4 must hold for a valid transaction.

Distributed Transactions

Transaction spans multiple data repositories on different nodes. Challenge: ensure all-or-nothing across all sites despite machine, software, and network failures.

Uses a Transaction Coordinator (one site nominated as coordinator) and Transaction Managers at each participating site.

Two-Phase Commit (2PC) ⭐

Ensures atomicity across all sites.

Phase 1 – Voting:
1. Coordinator writes "Prepare T" to its log, sends prepare T to all sites
2. Each site works on its component
3. Site ready → writes "Ready T" to log, sends Ready T to coordinator (enters pre-committed state, holds locks)
4. Site not ready → writes "Don't Commit T", sends Don't Commit T

Phase 2 – Decision:
1. If coordinator received Ready T from ALL → writes "Commit T", sends commit T to all
2. If any site sent Don't Commit T → writes "Abort T", sends abort T to all
3. Sites receiving commit T: commit, release locks, send Done
4. Sites receiving abort T: rollback, release locks, send Done
In 2PC, once a site sends "Ready T" it CANNOT abort on its own — only the coordinator can decide.

1. Master-Slave vs P2P Models

Distributed System Models

  • Architectural Models: How components are placed/organized (P2P or Client-Server)
  • Interaction Models: How time is handled (synchronous vs asynchronous)
  • Fault Models: What kinds of faults can occur (omission, arbitrary, timing)

P2P (Peer-to-Peer) Architecture

  • Every node acts as both client and server
  • No hierarchy, no single coordinator, no global view of system
  • Self-organizing, distributed control
  • Pros: Scalable, no SPOF, resilient to failures, no performance bottleneck in principle
  • Cons: Hard to manage, security challenging, decision making harder

Master-Slave (Client-Server) Architecture

  • Nodes are unequal — master coordinates slaves
  • Master: central coordinator, makes decisions
  • Pros: Easy monitoring, easier consistency/linearizability
  • Cons: SPOF at master, performance bottleneck as workers scale
P2PMaster-Slave
SPOFNoYes (master)
ScalabilityIndefiniteLimited by master
SecurityHarderEasier to control
Decision makingHarderEasy (central)
Real-world exampleAmazon Aurora multi-masterHadoop MapReduce

Eucalyptus (Cloud Controller Context)

  • CLC (Cloud Controller): High-level resource tracking, only ONE per cloud, maintains System Resource State (SRS)
  • Cluster Controller: Manages node controllers, deploys VMs
  • Node Controller: Runs on VM host, interacts with OS/hypervisor
  • On user request: SRS checked → resources reserved → request sent to CC → NC → on success committed, else rolled back

2. Unreliable Communication & Fault Models

Challenges with Distributed Communication

A request may: be lost, wait in queue, be processed but response lost, remote node failed, remote node slow/temporarily stopped. Non-deterministic + partial failures = core difficulty.

Fault Terminology

  • Fault: Root cause (e.g., hardware defect)
  • Error: Fault manifests in system state
  • Failure: System stops providing required service
  • Transient: Appears once, disappears
  • Intermittent: Comes and goes unpredictably (worst kind)
  • Permanent: Only fixed by replacing/repairing the component
MTBF = Total uptime / Number of breakdowns

Fault Models

ModelDescription
Crash-stopNode stops responding forever at some point. May or may not be detectable.
Crash-recoveryNode crashes and restarts after unknown time. Stable storage preserved; in-memory state lost.
Byzantine (arbitrary)Node behaves arbitrarily — may send malicious or incorrect messages.
Omission faultsMessages silently dropped
Timing faultsResponse outside expected time bounds

Network Issues

  • Timeouts: No correct value — measure distribution of round-trip times experimentally; use adaptive timeouts based on observed jitter
  • Congestion/Queueing: Busy links → packets wait; switch queues fill; virtualized multi-tenant networks add unpredictable delays
  • Noisy neighbor: Other tenants using shared resources cause variable delays

Failure Detection

  • Heartbeats (Push model): Process q periodically sends "I am alive" to monitor p. If p doesn't receive after timeout T, suspects failure.
  • Probing (Pull model): Monitor p sends "are you alive?" to q; q responds. On no response after T, p suspects failure.
  • Dual scheme: Combine both — more efficient
  • Trade-off: Low threshold → fast detection but more false positives. High threshold → fewer false alarms but slower detection.

Example: GFS uses centralised heartbeats — Master communicates with all chunkservers. On failure, master replicates data to another chunkserver.

Strategies After Detecting a Fault

  • Async communication: Avoid long chains of synchronous calls → use eventual consistency + event-driven
  • Retries with exponential backoff: Retry on transient failures; delay grows exponentially to avoid overload
  • Timeouts: Always set; prevent indefinite blocking
  • Circuit Breaker: After X failures, stop sending requests for a timeout period, then retry
  • Fallbacks: Return cached data or default value on failure
  • Bulkheads (Isolation): Separate thread/process pools per function — one failure doesn't bring down others
  • Failover: Switch to a replica (standby) on failure

Failover Architectures

  • Active-Active (symmetric): Both servers run; on failure, survivor takes all. Continuous availability.
  • Active-Passive (asymmetric): Primary active, secondary idle standby. On failure, secondary takes over. Failback = returning to primary.

3. Fault Tolerance & Availability

System Availability

Percentage of time a system is operational. 100% = never fails. "Five 9s" = 99.999% = ~5 min downtime/year.

Downtime can be from unplanned events (crashes) or planned maintenance. Slow/non-responsiveness is also a form of unavailability.

High availability = remove all SPOFs through redundancy + automatic failover.

Fault Tolerance

Ability to continue operating without interruption despite component failures. Can be hardware, software, or combined.

  • Fail-safe tolerance: Safety preserved but performance may drop (e.g., traffic lights → all red on failure)
  • Graceful degradation: App continues in reduced mode

Approaches to Build Fault Tolerance

  1. Redundancy: Eliminate SPOFs — duplicate power supplies, processors, memory, disks
  2. Reliability: Dependable components. Measured by MTTF (Mean Time To Failure) and MTTR (Mean Time To Repair)
  3. Recoverability: Retries, acknowledgements, ECC (Parity Bit, Hamming Code, CRC), Checkpointing + Rollback
MTTF = Average time before first failure | MTTR = Avg repair time
MTBF = MTTF + MTTR. Higher MTTF and lower MTTR = better availability.

4. Consensus & Paxos

The Consensus Problem

In a distributed system, N nodes must agree on a single output value despite failures. Each node has an input (0 or 1) and output (decided once, cannot change).

Requirements for a consensus algorithm:

  • Agreement: All non-faulty nodes decide on the same value
  • Termination: All non-faulty nodes eventually decide
  • Validity: The decided value was proposed by some node
  • Integrity: A node decides at most once

General Steps

  1. Elect: Elect a leader process to propose next value
  2. Vote: Non-faulty processes validate and vote on the proposed value
  3. Decide: If threshold votes agree, decide on that value; else repeat

Consensus in Sync vs Async Systems

  • Synchronous systems: Bounded message delays and processing times → consensus is solvable
  • Asynchronous systems (the Internet): Unbounded delays → consensus is impossible to guarantee (Fischer-Lynch-Paterson / FLP impossibility). Probabilistic solutions used in practice.
FLP theorem: Consensus is impossible in a purely asynchronous system with even one possible faulty process.

Paxos Algorithm ⭐

Family of distributed consensus algorithms used by Google (Chubby) and Amazon. Three roles: Proposers (leaders/coordinators), Acceptors (listen + respond), Learners (learn final decided value). A majority of acceptors must agree for a value to be accepted.

Paxos Steps

  1. Proposer sends prepare(n, v) to all acceptors (n = proposal number)
  2. Acceptor: if n > highest seen proposal number (nv), respond with ack(nv, v') where v' = value of highest accepted proposal
  3. Proposer gets acks from majority → sends accept(n, max(v, v' from acks))
  4. Acceptor: accepts value unless it has seen a higher n proposal. If majority accept → that value is decided. Learners are notified.

Problems Solved by Consensus

  • Reliable multicast (same updates in same order)
  • Membership/failure detection (group awareness)
  • Leader election
  • Mutual exclusion (distributed critical section)

5. Leader Election Algorithms

Why Leader Election?

Many distributed algorithms need a leader. If the leader fails, a new one must be elected. Goal: exactly one non-faulty process becomes leader; all agree on who it is.

When to elect: System start-up, current leader fails (no response within timeout).

Conditions: Liveness — every node eventually enters elected/not-elected state. Safety — only one node enters elected state.

Ring Election Algorithm

  1. N processes in a logical ring; messages go clockwise. Any process detecting leader failure sends an election message with its own ID.
  2. On receiving election message: if received ID > own ID → forward message. If received ID < own ID and haven't forwarded own election → overwrite with own ID and forward. If received ID == own ID → you are the new coordinator!
  3. Winner sends elected message around the ring; all processes set their leader to winner's ID.
Message Complexity = 3N − 1 (worst case)

Modified Ring Election

Each process appends its ID to message rather than overwriting. Initiator collects full list, picks best. Handles concurrent failures during election. Bypasses failed processes.

Bully Algorithm ⭐

Works in a fully-connected network where all processes know each other's IDs. Process with highest ID becomes leader (it "bullies" lower ones).

Steps

  1. Process p detects leader failure
  2. If p has highest known ID → declares itself leader, sends Coordinator to all lower IDs
  3. Else → sends Election to all processes with higher IDs
  4. If no response within timeout → p declares itself leader
  5. If a higher process responds with disagree/ok → wait for Coordinator message from that process; if none received, restart election
  6. Any process receiving Election message replies with ok/disagree and starts its own election
Bully Algorithm
Best case2nd highest process notices failure → sends N-2 Coordinator messages. Turnaround: 1 message transmission.
Worst caseLowest ID process initiates. N-1 processes start elections. Message overhead: O(N²). Turnaround: ~5 message transmissions.
First timeout = 2T (round trip time). Assumes synchronous system.
Bully: highest ID wins. Ring: messages travel around ring. Modified ring: appends IDs. Bully worst case = O(N²) messages.

6. Resource Management & Scheduling

Cloud Resource Management (CRM) Policies

  1. Admission control: Prevent accepting workload violating system policies
  2. Capacity allocation: Provision appropriate sizes for resources
  3. Load balancing: Distribute workload evenly
  4. Energy optimization: Minimize power consumption
  5. QoS guarantees: Meet SLA conditions
  6. Locale-based: Legal compliance, performance per location
  7. Cost: Auctions/perceived value

CRM Mechanisms

  • Control theory: Feedback-based (system utilization, latency) to control resource allocation
  • Utility-based: Performance models + SLA (rewards/penalties) guide allocation
  • Machine learning: Predict resource requirements from historic data
  • Market-oriented: Auction mechanisms (users bid for resources)

Scheduling Algorithms

  • Batch systems: Maximize throughput, minimize turnaround. Algorithms: Round-Robin, FCFS, SJF, Priority.
  • Real-time systems: Meet deadlines. Hard deadlines (strict, precise, penalties) vs Soft deadlines (guidelines, no penalties).

Workload Partition Rules

  • Optimal Partitioning Rule (OPR): Partition so all workers finish at the same time (earliest possible completion)
  • Equal Partitioning Rule (EPR): Assign equal workload to each worker

Scaling Strategies

  • Vertical scaling: Add more CPU/RAM to existing server (scale up)
  • Horizontal scaling: Add more servers, split workload (scale out)
  • Cloud bursting: Private cloud at 100% capacity → overflow to public cloud. Used in hybrid clouds.
  • Elasticity: Dynamic auto-scaling in real-time. Different from scalability (planned growth).

7. Distributed Locking & ZooKeeper

Distributed Locking

Mutual exclusion lock with global presence. Different clients on different machines must coordinate before accessing shared resources.

Why use it: Efficiency (avoid duplicate work), Correctness (prevent data corruption, lost updates, inconsistency).

Features of Distributed Locks

  • Mutual Exclusion: Only one client holds the lock globally at any moment
  • Deadlock-free: Lease-based mechanism — lock auto-expires if client crashes
  • Consistency: Lock remains consistent after failovers

Types

  • Optimistic locks: Proceed without blocking; check if record was modified before commit. If conflict → retry.
  • Pessimistic locks: Acquire exclusive lock before any operation; release after.

Fencing Tokens

Lock server assigns monotonically increasing fencing token with each lock grant. Write requests to storage must include fencing token. Storage rejects requests with outdated tokens. Prevents GC-paused clients from making stale writes after their lease expires.

Distributed Lock Manager (DLM) Implementations

  • Google Chubby, Apache ZooKeeper, Redis (Redlock algorithm)

Apache ZooKeeper ⭐

Open-source distributed coordination service. Provides naming, configuration management, distributed locks, synchronization, group services. Used by Kafka, HBase, Storm, Yarn.

Architecture

  • Server (ensemble/cluster): Multiple ZK servers. One elected as Leader at startup. All store copy of data tree in memory. Clients connect to any server.
  • Client: Sends requests + heartbeats to server. Auto-redirects on server failure.
  • Leader: Handles all write requests. Performs automatic recovery on node failure.
  • Follower: Handles read requests. Follows leader's instructions.

Data Model: znodes

Hierarchical namespace like a file system. Each znode has data + optional children. Limited to a few KB (coordination data only — status, config, location info). Entire tree in main memory.

  • Persistent znodes: Permanent; must be explicitly deleted. Survive session end.
  • Ephemeral znodes: Deleted when creating session ends. Used to detect client termination. Cannot have children.
  • Sequence znodes: Appends monotonically increasing counter to path name.

Operations & Consistency

  • Operations: create, delete, exists, getData, setData, getChildren, sync
  • Reads: Served locally from connected server (fast)
  • Writes: Forwarded to leader → goes through majority consensus (Zab protocol) → response sent
  • Watches: Clients set watches on znodes; notified on any change
  • All updates are totally ordered using zxid (ZooKeeper Transaction ID)

ZooKeeper Use Cases

  • Naming service: Maps names to service info; group membership
  • Distributed locks: Client creates ephemeral sequential znode; lowest ID gets the lock; others watch and wait
  • Leader election: Automatic fail-over with off-the-shelf support
  • Config management: Centrally stores config; new nodes pick it up on join
  • Message queue: znodes as queue items
ZooKeeper: reads are local, writes go through leader. Ephemeral znodes die with session. Zxid = total order for updates.

8. Reverse Proxy, Hybrid Cloud, Multitenancy & Security

Nginx architecture: Master-slave, async, non-blocking event-driven. Master assigns jobs to workers; each worker handles 1000+ connections in single-threaded manner. Benefits: load balancing, security (hides backend IPs), caching, centralized logging, TLS/SSL.

Multitenancy

Single software instance serves multiple customers (tenants). Enables better resource utilization and lower costs.

ArchitectureDescriptionTrade-off
Single multi-tenant DBOne app + one DB for all tenantsHighly scalable, low cost but complex to design
One DB per tenantOne app + separate DB per tenantSimpler operations, but less scalable & higher cost
Standalone single-tenantSeparate app + DB per tenantHighest isolation, highest cost

4 Levels: Ad hoc → Configurable → Configurable multi-tenant efficient → Scalable configurable multi-tenant (on cluster, unlimited scale).

Resource sharing: Table sharing vs dedicated tables. ACL-based or capability-based access control. Customization via pre-allocated columns, name-value pairs, or XML.

Cloud Security

Physical Security

Layered: central monitoring, threat detection (intrusion + natural hazards), staff training, backup systems, secure access.

Virtual Security Best Practices

  • Cloud Time Service (NTP): Sync clocks across datacenter for log correlation and correct operation
  • Identity Management: Scalable, federated (single sign-on), meets legal requirements, maintains audit trail
  • Access Management: Multi-factor auth for sensitive ops, no shared accounts, IP whitelisting
  • Break-glass procedures: Emergency bypass of security controls with alarm trigger
  • Key Management: Secure generation, assignment, revocation, archiving of encryption keys
  • Auditing: Captures all security events (time, system, user). Centralized, secure audit log.
  • Security Monitoring: Intrusion detection on network and hosts

Risk Management Process

  1. Categorize information resources by criticality + sensitivity
  2. Select appropriate security controls
  3. Assess if controls adequately address threats
  4. Implement security controls
  5. Monitor operational effectiveness
  6. Periodic review (new threats, operational changes)

Security Design Patterns

  • Defense in Depth: Layer defenses (VPN + IP whitelist + OTP). Attacker must break multiple layers.
  • Honeypots: Decoy systems to attract attackers and observe their behavior
  • Sandboxes: Execute software in restricted environment; limits damage if compromised
  • VM Isolation: Encrypt VM traffic, tighten port controls
  • Subnet Isolation: Physically separate admin, customer, and storage network traffic

DoS / DDoS / EDoS

  • DoS: Single source overwhelming target with requests. Buffer overflow or flood attacks.
  • DDoS: Distributed — many compromised bots (botnet) send requests. Hard to distinguish from normal traffic.
  • EDoS (Economic DoS): Exploits cloud auto-scaling. Attacker sends gradual fake traffic → triggers auto-scaling → victim's cloud bill explodes. More intelligent and sustained than DDoS.
EDoS targets the billing model of cloud, not just availability. DDoS = many bots from botnet.

Failure Detection & Application Recovery

  • Heartbeats: Instance sends periodic signal; silence → declared failed
  • Probes: Monitor sends lightweight request; no response → failed
  • Redirection: After failure detection, use HTTP 3xx redirects to route clients to healthy servers
  • Checkpoint/Restart: Periodically save application state. On failure, restart from last checkpoint. Distributed checkpointing: all instances checkpoint together.

Edge Computing (Brief)

  • Moves compute/storage closer to data source (IoT devices, gateways) rather than sending everything to cloud
  • Benefits: Ultra-low latency (single-digit ms), bandwidth optimization (filter/aggregate locally), enhanced privacy, reliability (works offline), cost efficiency
  • Challenges: Resource heterogeneity, security, programming complexity
  • vs Cloud: Edge = near data, real-time. Cloud = centralised, heavy compute.
  • Tools: Azure IoT Edge, AWS Greengrass, OpenStack (edge), EdgeX Foundry
